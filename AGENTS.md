## Learned User Preferences

- When asked to commit changes in this repo, default to committing AND pushing in the same step — that is the recurring ask.
- Run the site locally with a vendored bundle (`bundle config set --local path vendor/bundle && bundle install`, then `bundle exec jekyll serve`) before pushing; do not try to install gems globally on this machine.
- In this workspace, "Personalization" means Marketing Cloud Personalization (formerly Interaction Studio / Evergage); do not recommend Personalization-on-Core tools like the Sitemap Converter.
- `mcp/sitemap.js` is the versioned source of truth for the Personalization sitemap. When it changes, copy its entire contents into the MCP Visual Editor (Web → Sitemap → "Sitemap JS" tab) → Save → Execute (dry-run) → Publish; the Jekyll `jekyll-sitemap` plugin only governs the SEO XML sitemap. Reload in an incognito window after publish to bypass the Visual Editor sitemap cache.

## Learned Workspace Facts

- Repo is the Jekyll personal portfolio published at `bombonato.net` via GitHub Pages (`macielbombonato.github.io`); local toolchain is Ruby 2.6.10 + Bundler 1.17.2 with `vendor/`, `.bundle/`, `_site/`, `.jekyll-cache/` gitignored.
- Personalization beacon is loaded from `_includes/head.html` and `_includes/head-noads.html`, pointing to dataset `bombonato_prd`.
- Personalization content model: standard MCP `article` entity with two article types — `career` (career experiences) and `blog` (blog posts).
- Career content lives in `career/_posts/`, renders through `_layouts/career.html` + `_layouts/career-layout.html`. Front matter MUST include `industry`, `seniority`, `tags`, `technologies`, plus optional `end:` in ISO `YYYY-MM-DD` (empty = current job). Catalog `id` is auto-built as `{YYYYMM}-{slug}` from the post date so multiple roles at the same company get distinct items. The generated `catalog/experiences.json` is kept as backup; ingestion is Sitemap-only via beacon.
- Blog content lives in `blog/_posts/YYYY-MM-DD-Title.md`, renders through `_layouts/blog.html`. The index page `blog.html` uses `permalink: /blog/` so it appears in the Jekyll SEO sitemap. Front matter MUST include curated `description` and `tags` (auto-truncated content was unusable for the catalog). Both layouts emit `data-article-name` prefixed with `[Carreira]` or `[Blog]` so the MCP catalog UI groups items by type. The generated `catalog/blog.json` is kept as backup; ingestion is Sitemap-only.
- MCP catalog uses Sitemap-only ingestion. Critical setting: `Settings > Catalog and Profile Objects > Catalog Settings > Security > Enable Strict Catalog Security` MUST be UNCHECKED — when ON, beacon only registers item IDs and never writes attributes. The `commerce` gear's ETL (`CatalogObjectETL`) is SFTP/CSV-only (24-column schema with `id,categories,attribute:NAME,relatedCatalogObject:NAME`); per the official docs Sitemap and ETL must NOT be mixed on the same Object.
- DOM data attributes on the article wrappers carry per-page item context for the MCP sitemap to read: shared `data-article-id|name|category|url|author|publish-date|description|tags|topics`; career-only `data-article-company|start-date|end-date|location|industry|seniority|technologies`. Multi-value attrs (`tags`, `technologies`, `topics`) are comma-separated strings in the DOM and arrays in the JSON feed. Detail layouts expose `#mcp-related-careers` / `#mcp-related-blog` (with `data-mcp-zone`) which are now bound as ACTIVE contentZones (`related_careers`, `related_blog`) on `career_detail` and `blog_detail` pageTypes — Web Campaigns inject Templates into them. View events fire from `mcp/sitemap.js` via `SalesforceInteractions.sendEvent` with `CatalogObjectInteractionName.ViewCatalogObject` (avoid `Evergage.DisplayUtils.pageType` — not available in this MCP version).
- Pages `about.md` and `projects.md` were intentionally removed; the site only exposes Home, Carreira and Blog via `_includes/nav.html`. Do not reintroduce those routes.
- MCP Article Item Type custom attributes: `company` (String), `endDate` (Date, ISO `YYYY-MM-DD`), `industry` (String), `location` (String), `seniority` (String), `startDate` (Date, ISO `YYYY-MM-DD`), `technologies` (String, CSV), `topics` (MultiString — array of strings, NOT String). `tags` is NOT custom — it's the built-in polymorphic Tag slot; do not register a custom `tags` String (conflicts with built-in). Legacy duplicate items with slashed IDs (e.g. `/SAP/` vs `SAP`) created before commit `66d155b` need manual archive in the MCP Catalog UI.
- `topics` is the curated reportable taxonomy that complements free-form `tags`. Vocabulary lives in `_data/topics.yml` (key → PT-BR label) with 10 keys: `architecture`, `development`, `devops-cloud`, `security`, `marketing-tech`, `data`, `consulting`, `teaching`, `tooling`, `troubleshooting`. Each post carries `topics: [...]` (0–3 values). Layouts emit `data-article-topics` as CSV; the sitemap MUST map it via the `fromCsvAttr` helper (returns `string[]`) to match the MultiString slot — `fromSelectorAttribute` breaks ingestion. Bulk-update with `tools/apply_topics.py`. Use `topics` for MCP reports/segments (durable); `tags` for granular free-form labels.
- MCP sitemap & catalog payload rules: (a) every `pageType` MUST use `interaction:` (singular) — `interactions:` (plural) silently sends `interaction: null`, which MCP rejects with 400 and the browser surfaces as a CORS error; always include `pageTypeDefault` with a fallback interaction so unmatched pages never send null; use `matchWhenReady(selector)` for catalog pages so the `<article>` is in the DOM before `isMatch` runs; `career_detail` / `blog_detail` match selector is `article.post[data-article-category='career'|'blog']` (legacy `data-experience-*` was renamed to `data-article-*`). (b) Article has polymorphic typed collections needing a Jackson `"type"` discriminator with the MCP short id (NOT the class name): `Category` → `"c"`, `Tag` → `"t"`; shapes are `categories: [{"type":"c","_id":"blog","name":"Blog"}]` and `tags: [{"type":"t","_id":"git","name":"git"}]`; built-in collections go at TOP level of `catalogObject` while custom String/Date attrs go inside `catalogObject.attributes`; helper `fromCsvAttrAsTags(selector, attr)` in `mcp/sitemap.js` wraps CSV into Tag objects; wrong shape returns 400 (`missing type id property 'type'`, `Could not resolve type id 'X' ... known type ids = [c]`). Standalone items in dedicated feeds skip the discriminator because the feed targets the leaf type.
- MCP Web Campaign Templates (Web → Templates) are entities SEPARATE from Campaigns and have 4 tabs — `Handlebars` (markup), `CSS`, `Clientside Code`, `Serverside Code` (TypeScript via the Personalization Builder Templates SDK). `mcp/templates/` is the versioned source of truth: each template pairs `.hbs` (Handlebars) with `.ts` (Serverside Code); current set is `related_careers.{hbs,ts}` and `related_blog.{hbs,ts}`. Workflow on edit: paste each file into its tab in the MCP Template Editor. Recipes plug into Serverside Code via `RecommendationsConfig + recommend()` from the `recs` module; the marketer selects the Recipe per Campaign in the UI, so a single Template is reusable across Campaigns. MCP's Handlebars engine ONLY ships built-ins (`#if`, `#each`, `#with`, `#unless`, `lookup`, `log`) plus a few MCP helpers (`formatDate`, `formatCurrency`) — comparison helpers like `gt`, `lt`, `eq`, `and`, `or` are not registered and throw `ReferenceError: gt is not defined` at render; for "is array non-empty?" use `{{#if items.length}}` instead of `{{#if (gt items.length 0)}}`.
