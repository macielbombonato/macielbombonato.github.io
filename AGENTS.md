## Learned User Preferences

- When asked to commit changes in this repo, default to committing AND pushing in the same step — that is the recurring ask.
- Run the site locally with a vendored bundle (`bundle config set --local path vendor/bundle && bundle install`, then `bundle exec jekyll serve`) before pushing; do not try to install gems globally on this machine.
- In this workspace, "Personalization" means Marketing Cloud Personalization (formerly Interaction Studio / Evergage); do not recommend Personalization-on-Core tools like the Sitemap Converter.
- Sitemap configuration for Personalization is done in the MCP Visual Editor, not in code; the Jekyll `jekyll-sitemap` plugin only governs the SEO XML sitemap.

## Learned Workspace Facts

- Repo is the Jekyll personal portfolio published at `bombonato.net` via GitHub Pages (`macielbombonato.github.io`); local toolchain is Ruby 2.6.10 + Bundler 1.17.2 with `vendor/`, `.bundle/`, `_site/`, `.jekyll-cache/` gitignored.
- Personalization beacon is loaded from `_includes/head.html` and `_includes/head-noads.html`, pointing to dataset `bombonato_prd`.
- Personalization content model: standard MCP `article` entity with two article types — `career` (career experiences) and `blog` (blog posts).
- Career content lives in `career/_posts/`, renders through `_layouts/career.html` + `_layouts/career-layout.html`. Career front matter MUST include `industry`, `seniority`, `tags`, `technologies`, plus optionally `end:` in ISO `YYYY-MM-DD` (empty = current job). The generated `catalog/experiences.json` is kept as backup; current ingestion is Sitemap-only via beacon.
- Blog content lives in `blog/_posts/YYYY-MM-DD-Title.md`, renders through `_layouts/blog.html`. The index page `blog.html` uses `permalink: /blog/` so it appears in the Jekyll SEO sitemap. Blog front matter MUST include `description` (curated short summary) and `tags` — auto-truncated content was unusable for the catalog. The generated `catalog/blog.json` is kept as backup; current ingestion is Sitemap-only via beacon.
- MCP catalog uses Sitemap-only ingestion. Critical setting: `Settings > Catalog and Profile Objects > Catalog Settings > Security > Enable Strict Catalog Security` MUST be UNCHECKED — when ON, beacon only registers item IDs and never writes attributes. The `commerce` gear's ETL (`CatalogObjectETL`) exists but is SFTP-only and CSV-only (24-column schema with `id,categories,attribute:NAME,relatedCatalogObject:NAME` columns); per the official docs Sitemap and ETL must NOT be mixed on the same Object.
- DOM data attributes on the article wrappers carry per-page item context for the MCP sitemap to read: shared attrs `data-article-id|name|category|url|author|publish-date|description|tags`; career-only attrs `data-article-company|start-date|end-date|location|industry|seniority|technologies`. Multi-value attributes (`tags`, `technologies`) are comma-separated strings in the DOM and arrays in the JSON feed. `View Article` events fire from a sitemap JS hosted in the MCP Visual Editor, which reads those attributes and calls `SalesforceInteractions.sendEvent` with `CatalogObjectInteractionName.ViewCatalogObject` (avoid `Evergage.DisplayUtils.pageType` — not available in this MCP version).
- Pages `about.md` and `projects.md` were intentionally removed; the site only exposes Home, Carreira and Blog via `_includes/nav.html`. Do not reintroduce those routes.
- MCP Article Item Type has 8 custom attributes (all `String` except Dates): `company`, `endDate` (Date, ISO `YYYY-MM-DD`), `industry`, `location`, `seniority`, `startDate` (Date, ISO `YYYY-MM-DD`), `technologies` (CSV string). `tags` is NOT custom — it's the built-in polymorphic Tag slot; do not register a custom `tags` String (conflicts with the built-in). Legacy duplicate items with slashed IDs (e.g. `/SAP/` vs `SAP`) created before commit `66d155b` need manual archive in the MCP Catalog UI.
- MCP Article has polymorphic typed collections that all need a Jackson `"type"` discriminator with the MCP-specific short id (NOT the class name). Known mappings: `Category` → `"c"`, `Tag` → `"t"`. Working shapes are `categories: [{"type":"c","_id":"blog","name":"Blog"}]` and `tags: [{"type":"t","_id":"git","name":"git"}]`. Sending raw strings or wrong type ids returns 400 like `missing type id property 'type'` or `Could not resolve type id 'X' ... known type ids = [c]`. Apply in both feed JSON and sitemap JS; standalone items in dedicated feeds (e.g. `catalog/categories.json`) do not need the discriminator because the feed itself targets the leaf Item Type. Custom string attributes (industry, seniority) and custom string-array attributes (technologies) stay as raw strings — only the built-in typed collections need the wrapping.
- MCP sitemap `catalogObject` structure: built-in polymorphic collections (`categories`, `tags`) go at TOP LEVEL; custom String/Date attributes go inside `catalogObject.attributes`. Helper `fromCsvAttrAsTags(selector, attr)` (custom arrow fn) wraps CSV tags into `[{type:'t',_id:X,name:X}]` Tag objects. Custom String attrs use `SalesforceInteractions.resolvers.fromSelectorAttribute` directly with CSV strings (no transformation needed since the Item Type slot is typed String).
- MCP `career_detail` page match uses selector `article.post[data-article-category='career']` (the legacy `data-experience-*` attributes were renamed to `data-article-*`); update the sitemap in the MCP Visual Editor and reload the page in an incognito window after publish to bypass the Visual Editor sitemap cache.
