## Learned User Preferences

- When asked to commit changes in this repo, default to committing AND pushing in the same step — that is the recurring ask.
- Run the site locally with a vendored bundle (`bundle config set --local path vendor/bundle && bundle install`, then `bundle exec jekyll serve`) before pushing; do not try to install gems globally on this machine.
- In this workspace, "Personalization" means Marketing Cloud Personalization (formerly Interaction Studio / Evergage); do not recommend Personalization-on-Core tools like the Sitemap Converter.
- `mcp/sitemap.js` is the versioned source of truth for the Personalization sitemap. When it changes, copy its entire contents into the MCP Visual Editor (Web → Sitemap → "Sitemap JS" tab) → Save → Execute (dry-run) → Publish; the Jekyll `jekyll-sitemap` plugin only governs the SEO XML sitemap. Reload in an incognito window after publish to bypass the Visual Editor sitemap cache.

## Learned Workspace Facts

- Repo is the Jekyll personal portfolio published at `bombonato.net` via GitHub Pages (`macielbombonato.github.io`); local toolchain is Ruby 2.6.10 + Bundler 1.17.2 with `vendor/`, `.bundle/`, `_site/`, `.jekyll-cache/` gitignored.
- Personalization beacon is loaded from `_includes/head.html` and `_includes/head-noads.html`, pointing to dataset `bombonato_prd`.
- Personalization content model: standard MCP `article` entity with two article types — `career` (career experiences) and `blog` (blog posts).
- Career content lives in `career/_posts/`, renders through `_layouts/career.html` + `_layouts/career-layout.html`. Front matter MUST include `industry`, `seniority`, `tags`, `technologies`, plus optional `end:` in ISO `YYYY-MM-DD` (empty = current job). Catalog `id` is auto-built as `{YYYYMM}-{slug}` from the post date so multiple roles at the same company get distinct items. The generated `catalog/experiences.json` is kept as backup; ingestion is Sitemap-only via beacon.
- Blog content lives in `blog/_posts/YYYY-MM-DD-Title.md`, renders through `_layouts/blog.html`. The index page `blog.html` uses `permalink: /blog/` so it appears in the Jekyll SEO sitemap. Front matter MUST include curated `description` and `tags` (auto-truncated content was unusable for the catalog). Both layouts emit `data-article-name` prefixed with `[Carreira]` or `[Blog]` so the MCP catalog UI groups items by type. The generated `catalog/blog.json` is kept as backup; ingestion is Sitemap-only.
- MCP catalog ingestion is hybrid: (a) Sitemap beacon updates **attributes** + increments view counts on every page view; (b) `CatalogObjectETL` CSV feed (`catalog/articles.csv`, regenerated by `python3 tools/generate_catalog_feed.py`) bulk-writes the **`Article.categories` relation** — the beacon does NOT auto-link Articles to Categories, so without a feed commit recipes filtering on `Category = career|blog` return zero items and related-items campaigns render empty. Critical setting: `Settings > Catalog and Profile Objects > Catalog Settings > Security > Enable Strict Catalog Security` MUST be UNCHECKED — when ON, beacon only registers item IDs and never writes attributes. CSV ETL schema (per `CatalogObjectETL.ts`): MUST have `id` + `catalogObjectType` columns (the latter = `Article` for both career and blog — we use one Item Type and let `categories` differentiate). `categories` is a built-in MultiString column at TOP level (NOT `attribute:categories`); `attribute:topics` is MultiString with pipe `|` separator; `attribute:technologies` is plain String with comma `,` separator (NOT pipe — would store the literal pipe-delimited string and diverge from beacon data). The ETL rejects whole rows when an attribute column references an unregistered attribute — `author` and `publishDate` are NOT registered, use the System Attribute `published` (Date) instead. System Attributes available on Article: `name`, `url`, `imageUrl`, `description`, `promotable`, `archived`, `numRatings`, `rating`, `published`, `expiration`, `categories`. Production filename must start with `catalog-object-` to match the regex; the Gear Editor /testing/ path is dry-run only and never persists.
- DOM data attributes on the article wrappers carry per-page item context for the MCP sitemap to read: shared `data-article-id|name|category|url|author|publish-date|description|tags|topics`; career-only `data-article-company|start-date|end-date|location|industry|seniority|technologies`. Multi-value attrs (`tags`, `technologies`, `topics`) are comma-separated strings in the DOM and arrays in the JSON feed. Detail layouts expose `#mcp-related-careers` / `#mcp-related-blog` (with `data-mcp-zone`) which are now bound as ACTIVE contentZones (`related_careers`, `related_blog`) on `career_detail` and `blog_detail` pageTypes — Web Campaigns inject Templates into them. View events fire from `mcp/sitemap.js` via `SalesforceInteractions.sendEvent` with `CatalogObjectInteractionName.ViewCatalogObject` (avoid `Evergage.DisplayUtils.pageType` — not available in this MCP version).
- Pages `about.md` and `projects.md` were intentionally removed; the site only exposes Home, Carreira and Blog via `_includes/nav.html`. Do not reintroduce those routes.
- MCP Article Item Type custom attributes: `company` (String), `endDate` (Date, ISO `YYYY-MM-DD`), `industry` (String), `location` (String), `seniority` (String), `startDate` (Date, ISO `YYYY-MM-DD`), `technologies` (String, CSV), `topics` (MultiString — array of strings, NOT String). `tags` is NOT custom — it's the built-in polymorphic Tag slot; do not register a custom `tags` String (conflicts with built-in). Legacy duplicate items with slashed IDs (e.g. `/SAP/` vs `SAP`) created before commit `66d155b` need manual archive in the MCP Catalog UI.
- `topics` is the curated reportable taxonomy that complements free-form `tags`. Vocabulary lives in `_data/topics.yml` (key → PT-BR label) with 10 keys: `architecture`, `development`, `devops-cloud`, `security`, `marketing-tech`, `data`, `consulting`, `teaching`, `tooling`, `troubleshooting`. Each post carries `topics: [...]` (0–3 values). Layouts emit `data-article-topics` as CSV; the sitemap MUST map it via the `fromCsvAttr` helper (returns `string[]`) to match the MultiString slot — `fromSelectorAttribute` breaks ingestion. Bulk-update with `tools/apply_topics.py`. Use `topics` for MCP reports/segments (durable); `tags` for granular free-form labels.
- MCP sitemap & catalog payload rules: (a) every `pageType` MUST use `interaction:` (singular) — `interactions:` (plural) silently sends `interaction: null`, which MCP rejects with 400 and the browser surfaces as a CORS error; always include `pageTypeDefault` with a fallback interaction so unmatched pages never send null; use `matchWhenReady(selector)` for catalog pages so the `<article>` is in the DOM before `isMatch` runs; `career_detail` / `blog_detail` match selector is `article.post[data-article-category='career'|'blog']` (legacy `data-experience-*` was renamed to `data-article-*`). (b) Article has polymorphic typed collections needing a Jackson `"type"` discriminator with the MCP short id (NOT the class name): `Category` → `"c"`, `Tag` → `"t"`; shapes are `categories: [{"type":"c","_id":"blog","name":"Blog"}]` and `tags: [{"type":"t","_id":"git","name":"git"}]`; built-in collections go at TOP level of `catalogObject` while custom String/Date attrs go inside `catalogObject.attributes`; helper `fromCsvAttrAsTags(selector, attr)` in `mcp/sitemap.js` wraps CSV into Tag objects; wrong shape returns 400 (`missing type id property 'type'`, `Could not resolve type id 'X' ... known type ids = [c]`). Standalone items in dedicated feeds skip the discriminator because the feed targets the leaf type.
- MCP Web Campaign Templates (Web → Templates) are entities SEPARATE from Campaigns and have 4 tabs — `Handlebars` (markup), `CSS`, `Clientside Code` (JavaScript that runs in the browser), `Serverside Code` (TypeScript via the Personalization Builder Templates SDK). `mcp/templates/` is the versioned source of truth: each template ships as a triple `.hbs` + `.js` (Clientside Code) + `.ts` (Serverside Code); current set is `related_careers.{hbs,js,ts}` and `related_blog.{hbs,js,ts}`. We do NOT version `CSS` because the `.related-card` styles live in `assets/css/demo.css` and are shipped with the site, not with the template. Workflow on edit: paste each file into its matching tab in the MCP Template Editor.

- MCP Web Campaign Templates use a **MANUAL RENDERING strategy** in the `.js` (Clientside Code), NOT the MCP-managed `apply()` flow that injects the `.hbs` output into the Content Zone. Reason: on Rule Based Test campaigns the Content Zone Action could not be configured to reliably inject into our `<div id="mcp-related-careers"></div>` / `<div id="mcp-related-blog"></div>` targets — the server kept returning a full `campaignResponses[].payload.items` array but `#mcp-related-careers.children.length` stayed at 0 (silent failure, no error). The hook works by exploiting the bundle structure MCP delivers: `function registerTemplate(source) { ... }` is hoisted at the top of the bundle, and our IIFE (wrapped by MCP in `try { ... } catch(e)`) runs in the same scope, so we **reassign** `registerTemplate` to a wrapper that captures `source.items` and renders the cards ourselves into the target zone. We intentionally do NOT chain the original implementation — calling it would let MCP's `apply()` also fire (whenever it eventually works) and risk a duplicate render. As a consequence the `.hbs` files are now **design-only** mirrors of the card DOM (kept in sync with `.js` so the markup contract stays reviewable); the actual render path is the `buildCardHtml` function in each `.js`. Touch `.hbs` and `.js` together or styles drift. The MCP Date attributes arrive as Unix-ms numbers (e.g. `1738368000000`), so `.js` does its own PT-BR formatting via a local `MONTHS_PTBR` array (it does NOT depend on the MCP Handlebars `formatDate` helper because Handlebars is no longer in the render path); all injected strings go through `escapeHtml`/`escapeAttr` because we build innerHTML manually. Attribute access uses a `readAttr(attr)` helper that returns `attr.value` defensively (catalog payloads ship attributes as `{value, metadata}` objects). The blog `.js` reads the System Attribute `published` (NOT `publishDate` — unregistered, see CSV ETL fact above) with a fallback to `publishDate` for legacy payloads.

- The `.hbs` and `.js` are tightly coupled — the `.js` reads `data-cy-track` / `data-cy-target-id` from cards it ITSELF renders (mirroring the `.hbs`), so if you rename a selector in one you MUST rename it in the other. The `.js` fires interactions on top of the sitemap's `View Catalog Object`: `View Related Careers` + `Click Related Career` for the careers widget, `View Related Blog` + `Click Related Blog` for the blog widget — all wrap `SalesforceInteractions.sendEvent` in `try/catch` because tracking failures must never break the rendered widget or block navigation. Use per-card `addEventListener`, not a document-level delegated listener — MCP can re-render the same Campaign multiple times in one session and per-card listeners die cleanly with the DOM. Three Clientside Code design rules MUST hold or the MCP beacon's client-side rate limiter (logs `"Client: Event Rate Limiter triggered"` and silently drops the event) trips: (a) DISTINCT `interaction.name` per widget — both widgets coexist on detail pages and two back-to-back `sendEvent` with the same name in <1s gets throttled; do NOT use a generic shared name like `"View Recommendations"` for both; (b) PER-PAGE-LOAD de-dup via `window.__mcp_impressed_<widget>` flag — without it, every Campaign re-render (Auto Render in Template Editor, MCP decision refreshes) re-fires the same impression and trips the limiter; (c) RENDER-GUARD on `items.length > 0` before firing the impression — with manual rendering this is now a hard guarantee inside `renderCards`, but the check stays as defense-in-depth. Click handlers don't need de-dup because they only fire on actual user interaction. Recipes plug into Serverside Code via `RecommendationsConfig + recommend()` from the `recs` module; the marketer selects the Recipe per Campaign in the UI, so a single Template is reusable across Campaigns. ALWAYS chain `.restrictItemType("Article")` on `new RecommendationsConfig()` — without it the Campaign editor exposes an Item Type dropdown that defaults to the MCP placeholder `"Product"`, the Recipe dropdown then shows `"No options"` (catalog only has `Article`), the marketer cannot pick a Recipe, and at runtime `recommend()` is called with `recipeId: null` and throws `"System service exception via [recommend : context.services.recommendations.recommend(request)]"`. (Commit `83d7863` removed this call based on the hypothesis that it was throwing at runtime — that was a misdiagnosis; the actual root cause was elsewhere — Recipe unpublished, Strict Catalog Security on, or CSV ETL not yet run.) ALSO ALWAYS wrap `recommend(context, this.recsConfig)` in `try/catch` returning `{ items: [] }` on error — even with the Item Type locked, the Campaign can still be saved before a Recipe is picked, the selected Recipe can reference missing catalog data, or Strict Catalog Security can be re-enabled; the empty array + `{{#if items.length}}` guard + `.related-articles:has(.related-grid:not(:empty))` CSS rule hide the zone cleanly so the page stays presentable when the recommendations service fails. MCP's Handlebars engine ONLY ships built-ins (`#if`, `#each`, `#with`, `#unless`, `lookup`, `log`) plus a few MCP helpers (`formatDate`, `formatCurrency`) — comparison helpers like `gt`, `lt`, `eq`, `and`, `or` are not registered and throw `ReferenceError: gt is not defined` at render; for "is array non-empty?" use `{{#if items.length}}` instead of `{{#if (gt items.length 0)}}`.
- MCP Web Campaign modes: a campaign listed as `Published` can still be in `TEST MODE` (rule-based campaigns ship in test mode by default — the Campaign editor sidebar shows the Campaign Type literally as `Rule Based Test`, vs `Rule Based` for a live one, and a `+ 10% Control` line is the giveaway that a 10% slice of visitors is locked out of seeing anything regardless of mode). Two `userGroup` fields appear in the beacon response and they mean different things: the TOP-LEVEL `userGroup` (sibling of `campaignResponses`) reflects the visitor's own profile group (`"Default"` for anonymous public traffic); the NESTED `payload.userGroup` inside each `campaignResponses[]` entry reflects the CAMPAIGN'S mode (`"Test"` while in test, `"Default"` after promotion). The bundle and items still come back even in TEST MODE (debug sessions can see them), but the bundled `apply()` refuses to inject into the DOM when `payload.userGroup === "Test"` and the visitor is not flagged as a tester — so production users see an empty `#mcp-related-careers` / `#mcp-related-blog`. Promote via Campaign editor → `End Test` / `Promote to Production` (location varies by MCP UI version: top-right next to Publish, or the `⋯` Actions menu). Canonical post-promote smoke test in an incognito tab on a `career_detail` / `blog_detail` URL: confirm `payload.userGroup` flipped to `"Default"` in the Network tab, then run `document.querySelector('#mcp-related-careers').children.length` (and `#mcp-related-blog`) in DevTools — both should return > 0. The Visual Editor preview, Template Editor "TEST" button, and signed-in MCP admin sessions all bypass this gate and are NOT sufficient validation.
