## Learned User Preferences

- When asked to commit changes in this repo, default to committing AND pushing in the same step — that is the recurring ask.
- Run the site locally with a vendored bundle (`bundle config set --local path vendor/bundle && bundle install`, then `bundle exec jekyll serve`) before pushing; do not try to install gems globally on this machine.
- In this workspace, "Personalization" means Marketing Cloud Personalization (formerly Interaction Studio / Evergage); do not recommend Personalization-on-Core tools like the Sitemap Converter.
- `mcp/sitemap.js` is the versioned source of truth for the Personalization sitemap. When it changes, copy its entire contents into the MCP Visual Editor (Web → Sitemap → "Sitemap JS" tab) → Save → Execute (dry-run) → Publish; the Jekyll `jekyll-sitemap` plugin only governs the SEO XML sitemap. Reload in an incognito window after publish to bypass the Visual Editor sitemap cache.

## Learned Workspace Facts

- Repo is the Jekyll personal portfolio published at `bombonato.net` via GitHub Pages (`macielbombonato.github.io`); local toolchain is Ruby 2.6.10 + Bundler 1.17.2 with `vendor/`, `.bundle/`, `_site/`, `.jekyll-cache/` gitignored.
- Personalization beacon is loaded from `_includes/head.html` and `_includes/head-noads.html`, pointing to dataset `bombonato_prd`.
- Personalization content model: TWO Item Types — custom-configured `Article` for career experiences (28 items) and native `Blog` / `EVGBlog` for blog posts (25 items). The Item Type itself is the discriminator; recipes only need `restrictItemType("Article")` or `restrictItemType("Blog")` with NO Include Rule on `Category`. Replaced the previous single-`Article`+polymorphic-`categories` layout in commit `063d73a` and follow-up.
- Career content lives in `career/_posts/`, renders through `_layouts/career.html` + `_layouts/career-layout.html`. Front matter MUST include `industry`, `seniority`, `tags`, `technologies`, plus optional `end:` in ISO `YYYY-MM-DD` (empty = current job). Catalog `id` is auto-built as `{YYYYMM}-{slug}` from the post date so multiple roles at the same company get distinct items. The generated `catalog/experiences.json` is kept as backup; ingestion is Sitemap-only via beacon.
- Blog content lives in `blog/_posts/YYYY-MM-DD-Title.md`, renders through `_layouts/blog.html`. The index page `blog.html` uses `permalink: /blog/` so it appears in the Jekyll SEO sitemap. Front matter MUST include curated `description` and `tags` (auto-truncated content was unusable for the catalog). Both layouts emit `data-article-name` as the raw post title with NO `[Carreira]` / `[Blog]` prefix — the Item Type itself groups items in the MCP Catalog UI. The legacy `catalog/blog.json` is kept as backup; ingestion is bulk-via-CSV (`catalog/blogs.csv`) + beacon updates.
- MCP catalog ingestion is hybrid: (a) Sitemap beacon updates **attributes** + increments view counts on every page view; (b) `CatalogObjectETL` CSV feeds (`catalog/articles.csv` for Item Type `Article`, `catalog/blogs.csv` for Item Type `Blog`, BOTH regenerated by `python3 tools/generate_catalog_feed.py`) bulk-create / bulk-update items so recipes have a populated pool from day one. The beacon alone cannot create items in their correct Item Type — only the feed can — so without a feed commit the Recipe pool is empty and `recommend()` throws. Critical setting: `Settings > Catalog and Profile Objects > Catalog Settings > Security > Enable Strict Catalog Security` MUST be UNCHECKED — when ON, beacon only registers item IDs and never writes attributes. CSV ETL schema (per `CatalogObjectETL.ts`): MUST have `id` + `catalogObjectType` columns (`Article` for career rows, `Blog` for blog rows — they are SEPARATE Item Types, NOT one with a polymorphic discriminator). NO `categories` column on either CSV (the legacy single-Article+`categories` discriminator was retired in commit `063d73a`). `attribute:topics` is MultiString with pipe `|` separator; `attribute:technologies` is plain String with comma `,` separator (NOT pipe — would store the literal pipe-delimited string and diverge from beacon data). The ETL rejects whole rows when an attribute column references an unregistered attribute — `author` and `publishDate` are NOT registered on `Article`, use the System Attribute `published` (Date) instead; on `Blog` use the native System Attribute `publishedDate` (the docs state explicitly that `publishedDate` "must be exclusively used for articles and blogs"). System Attributes on `Article`: `name`, `url`, `imageUrl`, `description`, `promotable`, `archived`, `numRatings`, `rating`, `published`, `expiration`, `categories`. System Attributes on native `Blog` (extra over Article): `subtitle`, `modifiedTime`, `publishedDate`; and native `Author` / `Keyword` Tag types are queryable via `tools.global.authors` / `tools.global.keywords`. Production filename must start with `catalog-object-` to match the regex; the Gear Editor /testing/ path is dry-run only and never persists.
- DOM data attributes on the article wrappers carry per-page item context for the MCP sitemap to read: shared `data-article-id|name|category|url|author|publish-date|description|tags|topics`; career-only `data-article-company|start-date|end-date|location|industry|seniority|technologies`. Multi-value attrs (`tags`, `technologies`, `topics`) are comma-separated strings in the DOM and arrays in the JSON feed. Detail layouts expose `#mcp-related-careers` / `#mcp-related-blog` (with `data-mcp-zone`) which are now bound as ACTIVE contentZones (`related_careers`, `related_blog`) on `career_detail` and `blog_detail` pageTypes — Web Campaigns inject Templates into them. View events fire from `mcp/sitemap.js` via `SalesforceInteractions.sendEvent` with `CatalogObjectInteractionName.ViewCatalogObject` (avoid `Evergage.DisplayUtils.pageType` — not available in this MCP version).
- Pages `about.md` and `projects.md` were intentionally removed; the site only exposes Home, Carreira and Blog via `_includes/nav.html`. Do not reintroduce those routes.
- MCP Item Type custom attributes — TWO Item Types:
  - `Article` (career): `company` (String), `endDate` (Date, ISO `YYYY-MM-DD`), `industry` (String), `location` (String), `seniority` (String), `startDate` (Date, ISO `YYYY-MM-DD`), `technologies` (String, CSV), `topics` (MultiString — array of strings, NOT String).
  - `Blog` (native): ONLY `topics` (MultiString) is custom — everything else (`name`, `url`, `description`, `publishedDate`, `subtitle`, `modifiedTime`, `categories`, `tags`) is System on the native `Blog` Item Type.
  - `tags` is NOT custom on EITHER Item Type — it's the built-in polymorphic Tag slot; do not register a custom `tags` String (conflicts with built-in). Legacy duplicate items with slashed IDs (e.g. `/SAP/` vs `SAP`) created before commit `66d155b` need manual archive in the MCP Catalog UI. Legacy `[Blog]`-prefixed items under Item Type `Article` (created before the split) should be archived/deleted from the MCP Catalog UI after the split is live — they are duplicates of the items in Item Type `Blog`.
- `topics` is the curated reportable taxonomy that complements free-form `tags`. Vocabulary lives in `_data/topics.yml` (key → PT-BR label) with 10 keys: `architecture`, `development`, `devops-cloud`, `security`, `marketing-tech`, `data`, `consulting`, `teaching`, `tooling`, `troubleshooting`. Each post carries `topics: [...]` (0–3 values). Layouts emit `data-article-topics` as CSV; the sitemap MUST map it via the `fromCsvAttr` helper (returns `string[]`) to match the MultiString slot — `fromSelectorAttribute` breaks ingestion. Bulk-update with `tools/apply_topics.py`. Use `topics` for MCP reports/segments (durable); `tags` for granular free-form labels.
- MCP sitemap & catalog payload rules: (a) every `pageType` MUST use `interaction:` (singular) — `interactions:` (plural) silently sends `interaction: null`, which MCP rejects with 400 and the browser surfaces as a CORS error; always include `pageTypeDefault` with a fallback interaction so unmatched pages never send null; use `matchWhenReady(selector)` for catalog pages so the `<article>` is in the DOM before `isMatch` runs; `career_detail` / `blog_detail` match selector is `article.post[data-article-category='career'|'blog']` (legacy `data-experience-*` was renamed to `data-article-*`). (b) Article has polymorphic typed collections needing a Jackson `"type"` discriminator with the MCP short id (NOT the class name): `Category` → `"c"`, `Tag` → `"t"`; shapes are `categories: [{"type":"c","_id":"blog","name":"Blog"}]` and `tags: [{"type":"t","_id":"git","name":"git"}]`; built-in collections go at TOP level of `catalogObject` while custom String/Date attrs go inside `catalogObject.attributes`; helper `fromCsvAttrAsTags(selector, attr)` in `mcp/sitemap.js` wraps CSV into Tag objects; wrong shape returns 400 (`missing type id property 'type'`, `Could not resolve type id 'X' ... known type ids = [c]`). Standalone items in dedicated feeds skip the discriminator because the feed targets the leaf type.
- MCP Web Campaign Templates (Web → Templates) are entities SEPARATE from Campaigns and have 4 tabs — `Handlebars` (markup), `CSS`, `Clientside Code` (JavaScript that runs in the browser), `Serverside Code` (TypeScript via the Personalization Builder Templates SDK). `mcp/templates/` keeps the source-of-truth for the Serverside Code (`.ts`) — that's the only tab still live for the related-items widgets. Workflow on edit: paste each `.ts` into the Serverside Code tab of its matching Template in the MCP UI.

- The related-items widgets (`related_careers`, `related_blog`) render via a **SITE-SIDE renderer at `assets/js/mcp-related-renderer.js`**, NOT via the MCP Template's Clientside Code and NOT via the MCP-managed `apply()` flow. The site renderer is loaded synchronously from `_includes/head.html` (for `layout: post`, i.e. blog posts) and `_includes/head-noads.html` (for `layout: career`, i.e. career posts), BEFORE the Personalization beacon — so its `window.fetch` and `XMLHttpRequest.prototype.send` hooks are armed when MCP fires its first request. When a response body contains `{ campaignResponses: [{ payload: { contentZone, items } }] }`, the renderer walks it and dispatches to a per-zone builder (`buildCareerCard` / `buildBlogCard`) that injects cards directly into `#mcp-related-careers` / `#mcp-related-blog`. The renderer ALSO fires the impression and click tracking events. As a consequence: (a) the Templates' `.js` (Clientside Code) MUST stay empty — leaving rendering code there risks a double-render if MCP's `apply()` ever starts working; (b) the `.hbs` files are kept in the repo as design-only mirrors of the card DOM so the markup contract is reviewable, but they are not in the runtime path; (c) the Template itself still needs to exist in MCP because that is what binds the Recipe and triggers the campaign response — the renderer captures the response, not the template output. Exposes `window.__mcpRender(body)` for console smoke-testing.

- Two earlier render strategies failed silently and are documented as anti-patterns: (1) **MCP's native `apply()` injecting `.hbs` into the Content Zone** — on Rule Based Test campaigns the Content Zone Action could not be configured to reliably inject into our custom `<div id="mcp-related-careers" class="related-grid">` target; server returned full `payload.items` array but the zone stayed empty, no error in the console. (2) **Overriding `registerTemplate(source)` from inside the Template's Clientside Code** to capture `source.items` and render manually — the MCP runtime sandboxes the bundle in a way that didn't reliably surface the override to the caller; sometimes fired, sometimes didn't, no observable pattern. The site-side renderer sidesteps both by intercepting at the HTTP layer (the only layer guaranteed to fire on every detail page-load).

- Site renderer (`assets/js/mcp-related-renderer.js`) implementation details: ES5 only (no arrow, no const/let, no template literals, no optional chaining) because the site historically targets older browsers; `looksLikeMcpUrl` filters by URL regex `/evergage|evgnet|personalization|interactions/i` to avoid mutating unrelated fetches; XHR hook handles both `responseType === "json"` (pre-parsed) and raw `responseText` (JSON.parse); `readAttr(attr)` is the defensive accessor for the MCP attribute shape (`{value, metadata}` objects); dates arrive as Unix-ms numbers (e.g. `1738368000000`) and MUST be formatted with **UTC accessors** (`d.getUTCMonth()`, `d.getUTCDate()`, `d.getUTCFullYear()`) — using local accessors caused early-of-month posts to shift one day and render the wrong month label for viewers west of UTC (a Feb 1 UTC post showed as "Jan" in São Paulo); all injected strings go through `escapeHtml`/`escapeAttr` because innerHTML is built manually; blog cards read the native `Blog` System Attribute `publishedDate` first with fallbacks to `published` then `publishDate` for any legacy payloads still in flight from before the Item Type split; career cards read the custom Article Date attrs `startDate` / `endDate`; if the MCP response arrives before DOMContentLoaded, `deferRender` waits and retries; debug buffer `window.__mcpResponses` collects every intercepted response (`{ts, zonesSeen, totalItems, rendered, body}`) for post-hoc inspection.

- Related-items cards are capped at **3 per zone** in TWO places (defense-in-depth — bump in both or the lower one wins): (a) Serverside Code `.ts` chains `.restrictMaxResults(3)` onto the `RecommendationsConfig`, which tells MCP to compute and transmit at most 3 items and locks the field in the Campaign editor (`maxResultsIsRestricted: true` in the payload); (b) site renderer (`assets/js/mcp-related-renderer.js`) has a `MAX_PER_ZONE = 3` constant and slices `items` before render. The slice is the safety net if the marketer raises the cap in the Campaign editor or if someone forgets to re-paste the `.ts` into MCP after a commit. Rationale for 3: the related aside sits below the Disqus block; long lists scroll the user away from the article footer and dilute attribution.

- COLD-LOAD GOTCHA in the site renderer: `window.__mcp_rendered_<zone>` MUST only be set AFTER drawing at least one card, NEVER on an empty response. MCP frequently returns `items: []` on the very first campaign request of a session because recipes like `Related Career Experiences` need at least one prior catalog view to recommend anything; if we lock the zone on that first empty response we ignore the SECOND response (populated, fired right after the page-view event lands) and the widget only renders after the user does a manual reload (cmd+r), which clears `window.*` and lets the now-populated first response of the new page-load render normally. Order inside `renderZone`: (a) find target / `deferRender` if missing, (b) `items.length === 0 → return 0` (pass-through, no flag), (c) `__mcp_rendered_<zone> ? return 0`, (d) set flag, render, fire impression. Whatever you do, never set the flag before the items check.

- The site renderer is included CONDITIONALLY in the heads via `{% if page.layout == 'post' %}` (in `head.html`) and `{% if page.layout == 'career' %}` (in `head-noads.html`) to avoid global `fetch`/`XHR` mutation on pages without the `#mcp-related-*` zones. Career posts have `layout: career` (`_layouts/career.html` → `_layouts/career-layout.html` → `head-noads.html`); blog posts have `layout: post` (`_layouts/post.html` → `_layouts/blog.html` → `head.html`). The blog INDEX (`blog.html` at repo root) uses `layout: default-material` and does NOT load the renderer.

- Three event-tracking design rules MUST hold or the MCP beacon's client-side rate limiter (logs `"Client: Event Rate Limiter triggered"` and silently drops the event) trips: (a) DISTINCT `interaction.name` per widget — both widgets coexist on detail pages and two back-to-back `sendEvent` with the same name in <1s gets throttled; do NOT use a generic shared name like `"View Recommendations"` for both — the live names are `View Related Careers`/`Click Related Career` and `View Related Blog`/`Click Related Blog`; (b) PER-PAGE-LOAD impression de-dup via `window.__mcp_impressed_<widget>` flag; (c) RENDER-GUARD on `items.length > 0` before firing the impression — `renderZone` returns 0 and never fires the beacon when items is empty. Click handlers don't need de-dup. Per-card `addEventListener` (not document-level delegation) keeps listeners dying cleanly with the DOM on re-render. Recipes plug into Serverside Code via `RecommendationsConfig + recommend()` from the `recs` module; the marketer selects the Recipe per Campaign in the UI, so a single Template is reusable across Campaigns. ALWAYS chain `.restrictItemType("Article")` (for `related_careers`) or `.restrictItemType("Blog")` (for `related_blog`) on `new RecommendationsConfig()` — without it the Campaign editor exposes an Item Type dropdown that defaults to the MCP placeholder `"Product"`, the Recipe dropdown then shows `"No options"`, the marketer cannot pick a Recipe, and at runtime `recommend()` is called with `recipeId: null` and throws `"System service exception via [recommend : context.services.recommendations.recommend(request)]"`. The restrict ALSO ties the Template to a single Item Type — changing a Recipe's Item Type without updating its Template's `restrictItemType()` makes that Recipe disappear from the Campaign editor's dropdown and breaks the live widget; the two MUST be migrated together. (Commit `83d7863` removed this call based on the hypothesis that it was throwing at runtime — that was a misdiagnosis; the actual root cause was elsewhere — Recipe unpublished, Strict Catalog Security on, or CSV ETL not yet run.) ALSO ALWAYS wrap `recommend(context, this.recsConfig)` in `try/catch` returning `{ items: [] }` on error — even with the Item Type locked, the Campaign can still be saved before a Recipe is picked, the selected Recipe can reference missing catalog data, or Strict Catalog Security can be re-enabled; the empty array + `{{#if items.length}}` guard + `.related-articles:has(.related-grid:not(:empty))` CSS rule hide the zone cleanly so the page stays presentable when the recommendations service fails. MCP's Handlebars engine ONLY ships built-ins (`#if`, `#each`, `#with`, `#unless`, `lookup`, `log`) plus a few MCP helpers (`formatDate`, `formatCurrency`) — comparison helpers like `gt`, `lt`, `eq`, `and`, `or` are not registered and throw `ReferenceError: gt is not defined` at render; for "is array non-empty?" use `{{#if items.length}}` instead of `{{#if (gt items.length 0)}}`.
- BEHAVIORAL RECIPES + INCLUDE RULES = SILENT FALLBACK BUG (the anti-pattern that drove the Item Type split). On any behavioral algorithm (Item Affinity, Co-View, Similar Items, "Recommended For You"), the Include Rule is applied as a POST-rank filter — the engine first computes the top N candidates from the affinity matrix, THEN drops items that don't match. If the source pool is unbalanced (career had ~3× the aggregated views of blog), the top N comes back full of the dominant class, the filter drops everything, AND the engine falls back to a "popular items" bucket that IGNORES the Include Rule entirely — so a "Related Blog Articles" recipe with `Include Rule: Category = blog` silently returned career items. Diagnostic check: run the Recipe's `Test/Preview` in the MCP UI with a known Profile ID — if the preview returns items with the wrong Category, the Recipe itself is broken, not the Template or Campaign. The fix is structural — split the Item Type so the engine never ranks the wrong class to begin with (`restrictItemType("Blog")` at the SDK level scopes candidates BEFORE the affinity computation, leaving no room for fallback drift). NEVER reintroduce a single Item Type with `categories` as the discriminator on a behavioral recipe. Featured / Manually Curated / Newest recipes do honor Include Rules pre-rank, so the old layout would have worked there — but `restrictItemType` is still strictly better because it removes a whole class of footgun.
- The `.related-articles` aside footer carries a discreet "Powered by" attribution: a `.related-attribution` `<p>` with Material icon `auto_awesome` + "Recomendações geradas com Salesforce Marketing Cloud Personalization", placed as the LAST child of the aside in both `_layouts/blog.html` and `_layouts/career.html` so the parent's `.related-articles:has(.related-grid:not(:empty))` reveal toggles the footnote together with the cards (no extra `:empty` handling needed). Visual identity in `assets/css/demo.css`: color `#90a4ae` (same blue-gray as the `.related-section-title` icons), 11px / weight 500 / letter-spacing `.3px` (mirrors `.related-card-meta` typography scale), icon at 14px in `#b0bec5`, 1px top border at 6% opacity, centered flex with 6px gap so icon + text stay grouped on narrow viewports. ANTI-PATTERN: do NOT reuse the `.related-section-title` class for this footer and do NOT place it at the TOP of the aside — that class is reserved for the H3 section headers (16px bold with section-icon spacing), and a top placement makes the disclaimer read as a third section header instead of a footnote.
- MCP Web Campaign modes: a campaign listed as `Published` can still be in `TEST MODE` (rule-based campaigns ship in test mode by default — the Campaign editor sidebar shows the Campaign Type literally as `Rule Based Test`, vs `Rule Based` for a live one, and a `+ 10% Control` line is the giveaway that a 10% slice of visitors is locked out of seeing anything regardless of mode). Two `userGroup` fields appear in the beacon response and they mean different things: the TOP-LEVEL `userGroup` (sibling of `campaignResponses`) reflects the visitor's own profile group (`"Default"` for anonymous public traffic); the NESTED `payload.userGroup` inside each `campaignResponses[]` entry reflects the CAMPAIGN'S mode (`"Test"` while in test, `"Default"` after promotion). The bundle and items still come back even in TEST MODE (debug sessions can see them), but the bundled `apply()` refuses to inject into the DOM when `payload.userGroup === "Test"` and the visitor is not flagged as a tester — so production users see an empty `#mcp-related-careers` / `#mcp-related-blog`. Promote via Campaign editor → `End Test` / `Promote to Production` (location varies by MCP UI version: top-right next to Publish, or the `⋯` Actions menu). Canonical post-promote smoke test in an incognito tab on a `career_detail` / `blog_detail` URL: confirm `payload.userGroup` flipped to `"Default"` in the Network tab, then run `document.querySelector('#mcp-related-careers').children.length` (and `#mcp-related-blog`) in DevTools — both should return > 0. The Visual Editor preview, Template Editor "TEST" button, and signed-in MCP admin sessions all bypass this gate and are NOT sufficient validation.
